WHAT DOES 'VAR' ACTUALLY DO?

Primarily, var lets javaScript know that you are declaring a variable in the current scope.  If we declare a variable without var, javaScript will look up the scope chain for that variable.  If it doesn’t find it, it will simply create the variable in the global context, on the window object.

function testVar() {
  withoutVar = 'hello';
  var withVar = 'hello';
}
 
testVar();
console.log(withoutVar); // 'hello'
console.log(window.withoutVar);  // 'hello'
console.log(withVar); // ReferenceError: withVar is not defined

The var keyword (line 3) created the variable within the function’s scope, making it inaccessible outside the function.  withoutVar was actually created as a property of the window object.

Strict Mode Caveat

Strict mode won’t allow these shenanigans.   It’s too easy to create globals accidentally when you forget var, so strict mode will throw an error.

function testVar() {
  'use strict';
  warning = 'uh oh'; // ReferenceError: warning is not defined
  window.warning = &quot;uh oh&quot;; // this works
}
 
testVar();

In strict mode, create globals by explicitly creating a property on the window object.

Var affects ‘delete’

Ruby

withoutVar = 100;
var withVar = 100;
 
console.log(window.withoutVar); //100
console.log(window.withVar); //100
 
console.log(delete withoutVar); //true
console.log(delete withVar); //false!!

Because this code is outside any function, both these variables are created as properties on the window object. But the variable set with var cannot be deleted using ‘delete’.

Var creates a non-configurable property.  We can take a peek behind the curtain using Object.getOwnPropertyDescriptor()

When i first started doing little javaScript code challenges and small programs, i could use var or not, and i didn’t worry about it too much, because the code seemed to work either way.

But this is one of those cases where javaScript’s permissiveness can get you into trouble.  It’s easy to forget to use var within a function scope, and create a global by mistake.

So, best to be explicit!
