YOU ARE HERE: 'THIS' AND CONTEXT IN JAVASCRIPT

The meaning of this in javascript is so baffling, I sometimes wonder if a special effort was made to confuse us, like a prank gone too far. I am herewith going to try, in one post, to untangle this rat king for good and all. In order to do that i have to touch on a lot of topics.  Brace yourself!

First things first

Basically, ‘this’ refers to the current execution context. If you just start writing javascript in a browser, the default global context is an object called window.

Open a javascript console and enter ‘this’. It will return the Window object.

If you expand that object, you’ll see that it is absolutely huge, and gives us methods like alert() and setTimout().   When we use alert(), we are using window.alert(), and you could type it that way if you wanted.  In fact, whatever function calls you can make without specifying an object are functions on the window object.  window.console.log(‘hello’) is totally legit.

Note: window is considered a ‘Host’ object.  It is provided by the browser, and not part of the ECMAScript standard.

In a browser, the default global context is ‘window’.

Take a look at this code:

JavaScript
1
2
3
4
var name = "heidi";
console.log(name); //"heidi"
console.log(window.name); //"heidi"
console.log(this.name); //"heidi"

Easy enough, right? Ok, let’s briefly review…

Closures

A closure is a function inside another function that retains access to the variables in the outer function.  We don’t need to get too deep into it here, but this behavior helps underscore why context can be so counterintuitive in javascript.

JavaScript

function outer() {
  var name = "freddy";
  function inner() {
    //name is accessible from in here
    console.log(name);
  }
  inner();
}
 
outer();

By the way, you can actually see which variables are coming to your function by means of the closure in Chrome web dev tools. In the below code, ‘debugger’ creates a break point where the program will pause execution.  That allows us to examine the state of variables at that point.

  }

Run the above code with dev tools open.  Make sure to select the ‘sources’ tab, and make sure your javascript is showing as the source.

From here, you can see the closure variables in the ‘scope’ panel on the right side.  Since we have paused execution on line 4, the variables we see are specific to the inner() function.


Closures work for variables but NOT for context

Alright, so we can access variables from the outer function, but what about this from the outer function? Do we get that too?

No. We do not.

We can test this with an object literal.  Can you guess what the output values will be below?

JavaScript
1
2
3
4
5

10
var obj = {
  outer: function() {
    console.log(this);
    function inner(){
      console.log(this);
    };
   inner();
  }
}
obj.outer();

Run this code and you will see that the outer this is the ‘obj’ object, while the inner this is the ‘window’ object.

So the inner function totally reverts to a default context. The fact that it is inside obj does not automatically grant it the context of obj.

If we wanted our inner function to have access to the outer ‘this’, we could simply set a variable in the outer function to the value of this.  That variable WILL make it into our inner function via a closure.

JavaScript

var obj = {
  outer: function() {
    console.log(this);
    var self = this;
    function inner(){
      console.log(self);
    };
   inner();
  }
}
 
obj.outer();

Sneaky, right?  But there is a better way.  First, more basics…

Javascript has first class functions

JS functions are ‘first class’, meaning they can do what any other data type can do.  They can be assigned to a variable, passed to a function as an argument, returned from a function, etc.

In JS, functions are promiscuous devils.  Example:

JavaScript

var obj1 = {
  name: "jesse",
  logMe: function() {
    console.log( this.name );
  }
}
 
var obj2 = {
  name: "veronica"
}
 
obj2.logMe = obj1.logMe;
obj2.logMe();  // "veronica"
obj1.logMe(); // "jesse"
 
console.log(obj1.logMe === obj2.logMe); // true

On line 12, we are taking a function from obj1 and copying it (its’ reference, really) into obj2.  When we call that function as a method of obj2 (line 13), it gets the context of obj2.  Line 16 shows that logMe() in both objects points to the same reference.

So, asking the generic question “what is this in the logMe function” doesn’t really make sense in javascript.  It depends HOW you call the function.  Even though obj1.logMe and obj2.logMe refer to the same function in memory, the context within it changes based on how the function is invoked.

So, The context of a function depends upon how it is invoked at execution time, not how or where it is defined.

One more example:

JavaScript

var obj = {
  numbers: [12, 3, 56, 7],
  firstName: 'joe',
  logMe: function() {
    this.numbers.forEach(function(n) {
      console.log(n);
      console.log(this.firstName);
      console.log(this);
    });
  }
}
 
obj.logMe();

On line 13, the logMe function is called as a method of obj, so this within logMe will be obj.  So far so good.

But inside the forEach callback function, our context reverts back to window, and we lose access to ‘firstName’.  This loss of expected context will happen to you CONSTANTLY when you write callback functions within objects in javascript.

This is because javascript handles function invocation differently than method invocation.  More on that after we see how…

BIND, CALL, and APPLY will save your life.

.bind() .call() and .apply() can be called on any function, and you need to understand them.  Let’s revisit our function stealing example:

JavaScript
1
2
3

12
13
var obj1 = {
  name: "jesse",
  logMe: function() {
    console.log( this.name );
  }
}
 
var obj2 = {
  name: "veronica"
}
 
var logObj2 = obj1.logMe.bind(obj2);
logObj2();  // "veronica"

Bind allows us to pass a context to a function in this format: AnyFunction.bind(context).  It returns a function, which we can assign a new name as above.  logObj2 above is basically the obj1.logMe function, but we have specified that we want this in the function to be obj2.

Bind is super useful for callback functions inside objects that need to access the object variables.

JavaScript
1

var obj = {
  numbers: [12, 3, 56, 7],
  firstName: 'joe',
  logMe: function() {
    this.numbers.forEach(function(n) {
      console.log(n);
      console.log(this.firstName);
    }.bind(this));
  }
}
 
obj.logMe();

This on line 8 is the this of the logMe function.  That value gets passed to the forEach() callback function via bind, allowing us access to all the obj variables.

We can use .bind() with a named callback function like so:

JavaScript

var obj = {
  title: 'garbage engineer',
  delayOutput: function() {
    setTimeout(this.output.bind(this), 3000);
  },
  output: function() {
    console.log(this.title);
  }
}

Without the .bind(this) on line 4, our output function would log undefined.  It’s another case where we’re using a callback function, so the context would just revert back to window unless we tell it otherwise.

.apply() and .call() also specify the context of a function, but they actually execute the function as well.  .bind() merely returns a function with the given context, but does not actually execute it.

JavaScript

    console.log( this.name );
  }
}
 
var obj2 = {
  name: "veronica"
}
 
obj1.logMe.call(obj2); // "veronica"

The only difference between .call() and .apply() is how you pass variables to the target function.  .call() takes a comma-delimited series of values after the context argument, while .apply() accepts an array.

JavaScript

function logNames(name1, name2) {
  console.log(name1 + " " + this.lastName);
  console.log(name2 + " " + this.lastName);
}
 
var zuckers = {
  lastName: "Zucker"
}
 
logNames.call(zuckers, 'Jim', 'John');
logNames.apply(zuckers, ['Jim', 'John']);

Lines 10 & 11 above do effectively the same thing.  It all depends how you want to provide your arguments.

Method Invocation vs. Function Invocation

Here is the syntactical difference:

JavaScript
1
2
3
4
employee.totalSalary(); // this is a method invocation
                        // (preceded by an object)
 
totalSalary(); // this is a function invocation

Here’s the difference in interpretation:

JavaScript
1
2
3
4
5
6
7
employee.totalSalary()
// is equivalent to:
employee.totalSalary.call(employee);
 
totalSalary();
// is equivalent to:
totalSalary.call(window);

So let’s think about how a callback function works.  Essentially we pass a function into another function.  That function argument will be called internally by the function we passed it to.

See if you can follow this example:

JavaScript

Array.prototype.myForEach = function(callback) {
  for (var i = 0, length = this.length; i < length; i++) {
    callback(this[i]);  // function invocation
  }
}
 
var obj = {
  first: "nancy",
  countNumbers() {
    var arr = [1, 2, 3];
    arr.myForEach(function(n) {
      console.log(n);
      console.log(this.first); // undefined
    });
  }
}
 
obj.countNumbers();

Now we are getting to the heart of why callback functions seem to lose context.  When we call myForEach on line 11, we are passing a function to Array.prototype.myForEach.  Internally, that calls our function using a function invocation (line 3).  Thats why lines 12 & 13 have the window context, because they are within a callback that has been called using function invocation.

When you fix the above code using bind(), you are not passing in the same callback function as before, but a new function returned by bind that has its this set to the supplied context.

JavaScript

Array.prototype.myForEach = function(callback) {
  for (var i = 0, length = this.length; i &lt; length; i++) {
    callback(this[i]);  // the callback is now a function
                        // with its 'this' bound to 'obj'
  }
}
 
var obj = {
  first: "nancy";,
  countNumbers() {
    var arr = [1, 2, 3];
    arr.myForEach(function(n) {
      console.log(n);
      console.log(this.first); // nancy
    }.bind(this));  // 'this' will be 'obj' here
  }
}
 
obj.countNumbers();
