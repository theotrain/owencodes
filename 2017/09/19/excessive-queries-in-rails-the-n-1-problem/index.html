EXCESSIVE QUERIES IN RAILS (SOLVING THE N + 1 QUERY PROBLEM)

Any time you have a query taking place inside a loop, it should make you a bit uncomfortable. Because of overhead in the database and the network, its more efficient to do one query that returns 50 rows, than 50 queries that return one row each.

 Because of ActiveRecord’s convenient object-oriented syntax, beginners may not always realize when they are executing a query, so let’s see how we can get into this “too many queries” mess.

Imagine an application containing a Post model, with an index action like this:

Ruby

2
3
4
5
class PostsController < ApplicationController
  def index
    @posts = Post.all
  end
end

In our index view we want to show all the post titles, but also show the categories that pertain to each individual post. Something like this:

Ruby
1
2
3
4
5
6
7
8

10
<% @posts.each do |post| %>
  <% post.categories.each do |category| %>
    <span class="badge"><%= category.name %></span>
  <% end %>
  
  
<h4><%= post.title %></h4>
 
  <%= post.description %>
<% end %>

Because post.categories on line 2 creates a new query for each post, this is a classic N + 1 query problem.  If we look at the server output when we load the page in a browser, we can see 3 queries. Not so bad, but thats because my database only has 2 posts.

Started GET “/posts” for 127.0.0.1 at 2017-09-19 09:20:42 -0500
Post Load (0.4ms) SELECT “posts”.* FROM “posts”
Category Load (6.6ms) SELECT “categories”.* FROM “categories” INNER JOIN “post_categories” ON “categories”.”id” = “post_categories”.”category_id” WHERE “post_categories”.”post_id” = ? [[“post_id”, 1]]
Category Load (0.4ms) SELECT “categories”.* FROM “categories” INNER JOIN “post_categories” ON “categories”.”id” = “post_categories”.”category_id” WHERE “post_categories”.”post_id” = ? [[“post_id”, 2]]

In the controller we loaded all the Posts (1 query), then in the index view, we had individual queries to grab the categories for each post (2 more queries). If i had 50 posts to iterate through, i would have 51 total queries in order to render this page!

Solving the Problem with “.includes()”

The trick (in this case) is to load BOTH the posts and the categories up front.  That way we can access cached information as needed, rather than generating a new query for each post.

Ruby
1
2
3
4
5
class PostsController> ApplicationController
  def index
    @posts = Post.includes(:categories)
  end
end

Now when we load the index page we get… oh. Still 3 queries.

Started GET “/posts” for 127.0.0.1 at 2017-09-19 10:12:09 -0500
Post Load (0.4ms) SELECT “posts”.* FROM “posts”
PostCategory Load (0.7ms) SELECT “post_categories”.* FROM “post_categories” WHERE “post_categories”.”post_id” IN (1, 2)
Category Load (0.5ms) SELECT “categories”.* FROM “categories” WHERE “categories”.”id” IN (2, 1)

Because posts and categories have a many to many relationship, it queries the join table as well. The point is though, that if i had 50 posts, i would still only have to do 3 queries.  It’s a very elegant solution, and i don’t need to make any changes to my view!

If you have multiple relationships to load at once, you can do so with

Ruby
1
Post.includes(:categories, :comments)

You can also use limit…

Ruby
1
Post.includes(:categories).limit(10)

and where to make queries more specific…

Ruby
1
Article.includes(:comments).where(comments: { visible: true })

That last example comes straight from the Rails Guides “Active Record Query Interface” guide section on eager loading.  A great place for more details.
